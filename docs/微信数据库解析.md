### 聊天记录存放逻辑

1. `message_0.db`存放的是近期的聊天数据其中每个聊天框（群聊、私聊、系统等）都是一个单独的表，命名规则为`Msg_<username的md5>`，例如

   ```sql
   select * from contact where nick_name like '%第一%'
   -- 返回的结果有 username = 1234567890@chatroom
   -- 进行md5编码为 <chat_md5>
   -- 那么其真实的表为 Msg_<chat_md5>
   ```

   **字段解析：**

   1. `username`：底层区分的id
   2. `local_type`：消息的类型
   3. `nick_name`：用户、群聊的名称（网名）
   4. `remark`：备注

2. `message_1.db`可以理解为历史记录，例如`message_0.db`只会记录到`2025\01\01`，那么`message_1.db`可能会记录`2024\06\01 - 2025\01\01`

### 图片真实位置解析

1. 先确定一条消息是否为图片

   ```sql
   -- local_type = 3 为图片
   SELECT
       local_id,
       server_id,
       local_type,
       create_time
   FROM Msg_<chat_md5>
   WHERE local_id = <local_id>;
   -- 返回 <local_id>	<server_id>	3	<create_time>
   ```

   当前这一步至少要拿到：

   1. `local_id`
   2. `server_id`
   3. `create_time`
   4. `local_type`

   其中：

   - `local_type = 3` 说明这是一条图片消息
   - `create_time` 后续用于判断图片大致落在哪个月份目录
   - `local_id + server_id + create_time + local_type` 是和资源表做精确关联时最稳的一组条件

2. 接下来拿着这些信息去`MessageResourceInfo`查

   ```sql
   -- 这里直接
   SELECT
       message_id,
       chat_id,
       sender_id,
       message_local_type,
       message_create_time,
       message_local_id,
       message_svr_id
   FROM message_resource.MessageResourceInfo
   WHERE message_local_id = <local_id>
      AND message_svr_id = <server_id>
      AND message_create_time = <create_time>
      AND message_local_type = 3;
   -- 返回 <message_id>	<chat_id>	<sender_id>	3	<create_time>	<local_id>	<server_id>
   ```

   这里说明：

   1. `message_resource.db` 是消息与资源之间的映射层
   2. `message_id = 1485` 是这条图片消息在资源库中的唯一标识
   3. `chat_id = 13` 对应的是当前会话
   4. `sender_id = 1` 对应发送者

   这里不要只用 `message_local_id` 做关联，最好把 `message_svr_id + message_create_time + message_local_type` 一起带上，否则理论上存在误命中的可能

3. 接下来使用`message_id`去查找`MessageResourceDetail`的信息

   ```sql
   SELECT
       mrd.resource_id,
       mrd.message_id,
       mrd.type,
       mrd.size,
       mrd.create_time,
       mrd.access_time,
       mrd.status,
       mrd.data_index
   FROM message_resource.MessageResourceDetail AS mrd
   WHERE mrd.message_id = <message_id>
   ORDER BY mrd.type;
   -- <resource_id>	<message_id>	131073	<主图大小>	<create_time>	0	1	0
   -- <resource_id>	<message_id>	262145	<缩略图大小>	<create_time>	0	1	0
   ```

   当前样本里可以确定：

   1. `131073` 是主图资源，大小 `70951`
   2. `262145` 是缩略图资源，大小 `4570`

   后续如果某条图片还存在 `_h.dat`，通常表示还有更大规格的图片资源

4. 对于当前这类图片，`Msg_*.packed_info_data` 已经可以直接给出图片文件基名，不再需要通过 `hardlink.db` 反推文件名

   ```sql
   SELECT
       local_id,
       hex(packed_info_data)
   FROM Msg_<chat_md5>
   WHERE local_id = <local_id>;
   -- 返回
   -- <local_id>	<packed_info_hex>
   -- 16进制转换后可提取出 32 位十六进制文件基名
   ```

   这段 `packed_info_data` 拆开之后，核心信息是：

   - `<file_base>`

   它就是这张图片的文件基名，基于这个基名可以直接得到：

   1. 主图：`<file_base>.dat`
   2. 缩略图：`<file_base>_t.dat`
   3. 高清图（如果存在）：`<file_base>_h.dat`

   因此对于“定位某一条图片消息对应的文件名”这件事，最短链路其实是：

   ```text
   Msg_* -> packed_info_data -> 文件基名
   ```

5. 接下来通过 `Msg_*` 表名后缀和 `create_time` 直接拼出文件路径

   已知：

   1. `Msg_<chat_md5>` 中的 `<chat_md5>` 就是当前会话的 `chat_md5`
   2. `create_time = <create_time>` 对应的月份目录是 `<YYYY-MM>`
   3. 第 4 步已经拿到文件基名 `<file_base>`

   因此对于当前机器，图片的实际路径规则为：

   ```text
   /Users/<你的用户名>/Library/Containers/com.tencent.xinWeChat/Data/Documents/xwechat_files/<当前账号>/msg/attach/<chat_md5>/<YYYY-MM>/Img/<file_name>
   ```

   这条图片的主图路径为：

   ```text
   /Users/<你的用户名>/Library/Containers/com.tencent.xinWeChat/Data/Documents/xwechat_files/<当前账号>/msg/attach/<chat_md5>/<YYYY-MM>/Img/<file_base>.dat
   ```

   同理可以直接得到：

   ```text
   /Users/<你的用户名>/Library/Containers/com.tencent.xinWeChat/Data/Documents/xwechat_files/<当前账号>/msg/attach/<chat_md5>/<YYYY-MM>/Img/<file_base>_t.dat
   /Users/<你的用户名>/Library/Containers/com.tencent.xinWeChat/Data/Documents/xwechat_files/<当前账号>/msg/attach/<chat_md5>/<YYYY-MM>/Img/<file_base>_h.dat
   ```

6. `hardlink.db` 在当前这类图片分析里不再作为主流程

   对当前这类图片而言，`Msg_*.packed_info_data` 已经足够给出文件基名，因此定位文件时不再需要通过 `hardlink.db` 反推。

   `hardlink.db` 更适合作为补充索引库使用，例如：

   1. 批量扫描某个月某个会话下有哪些图片文件
   2. 交叉验证某个文件名是否已经实际落盘

#### 当前关于图片存储的结论

1. `Msg_*` 负责记录“这是一条图片消息”
2. `message_resource.db` 负责记录“这条图片消息有哪些资源”
3. `packed_info_data` 在当前样本里已经能直接给出图片文件基名
4. 对当前这类图片，定位图片文件时不再需要通过 `hardlink.db` 反推文件名
5. 最终图片文件落在：

   ```text
   msg/attach/<chat_md5>/<YYYY-MM>/Img/
   ```

#### 注意事项

1. 找到 `.dat` 文件并不等于已经拿到可以直接打开的图片
2. 当前这些 `Img/*.dat` 文件并不是标准 JPEG/PNG/WebP 文件
3. 它们也不是简单改后缀就能打开的格式
4. 当前更像是微信桌面版自己的图片封装/缓存格式
5. 因此“定位图片文件”和“把 `.dat` 还原成标准图片”是两个不同的问题，前者目前已经基本打通，后者还需要继续分析微信的解码逻辑

#### 目前最短的图片定位链路

```text
Msg_* -> local_type=3
      -> packed_info_data 取文件基名
      -> Msg_* 表名后缀得到 chat_md5
      -> create_time 得到 YYYY-MM
      -> 拼接到 msg/attach/<chat_md5>/<YYYY-MM>/Img/<base>.dat
```
